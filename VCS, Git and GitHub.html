Introduction

Version Control Systems (VCS)

What is a Version Control System (VCS)?

Version control is a system that records changes to a file or set of files over time so that you can recall specific versions later About Version Control, Pro Git book

Examples of simple VCS
  • Google Docs
  • Apple Time Machine
  • Ctrl+Z
  • Manually copy (i.e. You)
  • ...

Why use a VCS?

  • Record history
    Track the development of a project

  • Describe changes
    Find out why a specific change was made

  • Fix mistakes
    Retrieve code that was previously deleted or changed

  • Share updates
    Collaborate with other developers remoteley

  • Project managment
    Control project development and divide tasks

VCS generations

Features

Examples

First generation

  • Centralized on a single machine
  • No concurrent editing
  • File scope history
  • Source Code Control System (SCCS)
  • Revision Control System (RCS)

Second generation

  • Centralized on multiple machines
  • Merge before recording changes
  • File/set scope history
  • Changeset based
  • Concurrent Versions System (CVS)
  • Subversion (SVN)
  • Team Foundation Server

Third generation

  • Decentralized
  • Merge after recording changes
  • Fileset scope history
  • Changeset/Snapshot based
  • Bazaar
  • Git
  • Mercurial

Terminal

A terminal or text terminal is a program that allows us to interact with a computer with text rather than graphic user interfaces and with applications that have command line interfaces.

Commands

To run a command in a terminal just type it out and press enter. You might often see commands preceeded with $ or >. This are to indicate that the terminal is ready for a command but are not part of the command that you should type out and run.

Git commands

All git commands have the same format

$ git <git-command> [<value-1> <value-2> ...]

When you see an expression like <value> you should replace it and the brackets < and > with an appropriate value which will depend on the situation. An expression like [optional] is optional. If you use it when running a command you should replace it with optional.

git-config

$ git config [--global] <name> <value>

Configure options that determine how Git runs.

--global Set options that affect all Git repositories on your computer

name Name of the option that you want to change

value Value that you want name to be changed to

Install Git

If you don't have Git already installed (run git --version to find out) the simplest way to do this on Mac and Windows is to head over to git-scm.com and download and install the program. If you are on a debian based operating system run apt-get install git.

Configure Git

We will tell Git your name and email address so that commits you make are attributed to you

$ git config --global user.email <your-email>

Create an empty Git repository or reinitialize an existing one

Directory Directory in which to create an empty Git repository

git-status

Show the working tree status

Path Path to file or directory to check the status of. If you omit Path then the status of your entire repository will be checked.

Repository

A repository is your projects folder plus a special folder .git that is created during initialization inside your projects folder. This special folder stores all Git information including previous versions.

Commit

A commit is a snapshot of the files in your projects folder that is recorded by Git along with metadata including when it was committed and who made it.

Working tree

Your working tree are all the files in your repository other then the .git folder. We say that these files are checked out.

Index

A snapshot of your working tree that has not been commited. We can tell Git which modifications to include in the index.

git-add

$ git add <path-list>

Add file content modifications to the index

path A list space separated paths to add to the index. These can be file paths or directory paths.

git-commit

$ git commit [-m <message>]

Record changes to the repository

-m <message> A message to describe your commit.

Branch

A branch is a sequence of commits representing a single line of development. We refer to the tip of the branch as the branch head and it points at a specific commit.

Remote

A remote or remote repository is a copy of your repository that you are watching or tracking. Your repository may contain commits that it doesn't have and it may contain commits your repository doesn't have.

New GitHub Repository

git-remote

$ git remote add <remote-name> <remote-url>

Manage set of tracked repositories

remote-name The name you want to give this tracked remote repository.

remote-url URL of the remote repository.

git-push

$ git push [-u] [<remote-name>] [<remote-branch>]

Update remote repositories along with associated commits

remote-name The name of the remote repository you want to update.

remote-branch Branch you want to send updates to from the current branch.

-u If both remote-name and remote-branch are given, Git will rememeber so that you can omit them in future pushes.

git-merge

Join two or more development histories together

fast-forward merge

A fast-forward merge can be performed when the branch head of the current branch is a direct ancestor of the branch head of the merging branch. In this case the branch head is simply moved forward.

  2. Clone the fork

  3. Create and edit a branch

  4. Push to the fork

  5. Create a pull request
    Initiate a merge without commit between two forks

  6. Fetch changes from upstream
    Update local repository with commits from the original

Fork

A fork is a personal copy of another user's repository that lives on your account. Forks allow you to freely make changes to a project without affecting the original. Forks remain attached to the original, allowing you to submit a pull request to the original's author to update with your changes. You can also keep your fork up to date by pulling in updates from the original. Fork, GitHub glossary

git-clone

$ git clone <url> [<directory>]

Clone a repository into a new directory

url URL to a remote repository.

directory Directory in which to clone the repository. If this is omitted then a new folder will be created with the same name as the repository and it will be cloned into this folder.

Remote-tracking branches

A remote branch is a branch that exist on a remote repository. A remote-tracking branch is a local branch that follows what the remote branch is doing. An example of a remote-tracking branch is origin/master.

Upstream branches

An upstream branch of a local branch is the default branch that will be merged into the local branch. The local branch is often called a tracking branch. When you clone a repository an origin remote is automatically added and the master branch is setup to track the origin/master remote-tracking branch.
